## Problem Statement

기존 React+Tailwind 웹앱 `bus-finder` (모현읍 버스 파인더, 외대 학생용 버스/셔틀 시간표)를 Flutter 모바일 앱으로 리빌드해 App Store + Play Store에 v1.0까지 공개 출시한다. 주 목적은 사용자 획득이 아니라:

1. Flutter 학습 + 포트폴리오 확장
2. AI 에이전트팀(Claude Code + gstack + 병렬 서브에이전트) 주도로 풀스택 배포까지 완주하는 경험
3. "혼자서 48시간 만에 MVP 출시" + "이후 프로덕션 레벨 아키텍처까지 밀기" 두 가지 스토리 확보

카카오맵/네이버지도가 해결하지 못하는 **교내 셔틀 정보 부재** 지점을 wedge로 잡고, 사용자에게는 시간표가 아닌 **"지금 버스 잡을 수 있는가"라는 의사결정**을 제공한다.

## What Makes This Cool

**"Will I make it?"** — hero interaction. 시간표 앱이 아니라 **의사결정 앱**.

사용자가 강의실에서 셔틀 정류장으로 걸어가는 중에 앱을 열면, 화면 중앙에 숫자 하나가 뜬다.

**v0.1 (A) 버전:**
> **87%** — 평소 걸음 속도면 다음 셔틀 탑승 가능
>
> 실패 시: 다음 셔틀은 18분 뒤 출발

**v1.0 (B) 버전 (광역버스 API 추가 후):**
> **87%** — 평소 걸음 속도면 다음 셔틀 탑승 가능
>
> 실패 시 대안: 7017번 광역버스 (4분 뒤 도착)

입력: 현재 GPS + 걷는 속도 추정 + 셔틀 스케줄 (+ v1.0부터 광역버스 ETA 폴백).
출력: **데이터가 아니라 결정**.

### 확률 계산 공식 (v0.1)

```
p(make_it) = max(0, min(1, 1 - (t_walk + t_buffer) / t_until_departure))

t_walk           = distance_to_stop_m / walking_speed_mps
walking_speed_mps = 1.3 (default)
                  = GPS delta 이동평균 (최근 30초, 세션 동안 업데이트)
t_buffer         = 30s (정류장 접근 + 대기 여유)
t_until_departure = next_departure_ts - now_ts (초)

GPS 부정확(accuracy > 50m) or 미허용:
  → walking_speed_mps = default
  → UI에 "GPS 부정확" 표시 뱃지
```

정확도 캘리브레이션은 v1.x에서 실측 데이터로 튜닝.

**실측 데이터 수집 메커니즘 (v1.0에서 도입):**
- 사용자 동의 하에 익명 이벤트 로그: `{predicted_p, actual_outcome, t_walk_estimated, t_walk_actual}` → Supabase 테이블 `probability_samples`에 저장.
- 동의 UI: 설정 화면에서 "정확도 개선에 기여하기" 토글 (기본 OFF).
- v1.1+에서 샘플 >500개 누적 후 walking_speed 기본값 + t_buffer 재튜닝.
- v0.1은 위 공식 고정, 텔레메트리 없음.

이게 Kakao Map/Naver Map이 원칙적으로 못 만드는 이유: 이들은 HUFS 교내 셔틀의 존재를 인지하지 않음. 공공 데이터가 아니기 때문. 이 빈틈이 프로젝트의 wedge이자 포트폴리오 내러티브의 핵심.

Flutter 애니메이션 서피스로도 훌륭 — 확률 링, 라이브 카운트다운, 폴백 카드. 30초 스크린 녹화로 LinkedIn/면접용 데모 완벽.

## Constraints

- **시간:**
  - v0.1 (A): **48시간 집중 코딩 + 최대 1주 외부 승인 대기**. 코드/Supabase/CI 작업은 48h 내, Apple Developer 계정 승인(신원 확인 1-3일) + TestFlight External Review(1-3일 비결정적)은 병렬로 진행. 즉 "주말에 끝난다"는 **코딩 자체**를 의미하며, 실제 설치 링크 확보는 7-10일 총 소요.
  - v1.0 (B): 2-4주 (Apple 공개 스토어 심사 1-7일 포함)
- **예산:**
  - Apple Developer Program: $99/년 (v1.0 공개 스토어 진입시 필수)
  - Google Play Console: $25 일회성
  - Supabase/Vercel/GitHub Actions: Free tier 내
- **인력:** 1인 + AI 에이전트팀
- **플랫폼:** iOS + Android 동시 출시 (Flutter 선택의 전제)
- **데이터 출처:**
  - 광역버스: 경기도 버스정보시스템 공공 API (data.go.kr)
  - 교내 셔틀: HUFS 홈페이지 스크래핑 (공식 API 없음)
- **언어:** 앱 UI는 한국어, 코드 주석/커밋/문서는 영어 (포트폴리오 국제 어필)

## Premises (확정)

- **P1. 2단계 출시 전략** — v0.1 = TestFlight Public Link + Play Internal Test Link. 코딩은 주말 48h, 승인 대기는 별도로 1-7일 (Apple Developer 계정 승인 + External Review 비결정적). v1.0 = App Store + Play Store 공개 출시 (2-4주, 공개 심사 1-7일 포함). 두 스토리 다 확보.
- **P2. 데이터/인프라** — 광역버스 공공 API + 셔틀 스크래핑. 백엔드 = Supabase (DB + pg_cron + Edge Functions, 스크래핑/데이터 제공 all in one) + Vercel (광역버스 공공 API 키 프록시 전용). 클라이언트는 직접 스크래핑/공공 API 호출 안 함.
- **P3. CI/CD Day 1** — 빈 Flutter 앱 → GH Actions + Fastlane → TestFlight + Play Internal까지 자동화를 기능 개발보다 먼저. Day 1 커밋에 초록 CI 뱃지.
- **P4. 커스텀 디자인** — Material 3 기본 테마 거부. Phase 5 전에 `/design-consultation` 스킬로 디자인 시스템 확정 후 `packages/ui_kit`에 구현.
- **P5. 위젯 v2 연기** — iOS Live Activity + Android Home Widget은 v2. v1은 앱 본체 완성도에 집중.
- **P6. AI 에이전트팀 주도** — Claude Code + gstack의 설계→단계→검증 사이클 기반. 본인은 아키텍처 선택 이유를 설명 가능해야 함 (맹목적 "만들어줘" 금지).
- **P7. v1 Scope 경계** — GPS는 "Will I make it?"를 위해 **포함**. LLM 대화형 플래너, 실시간 차량 위치, 푸시 알림, 회원가입, 소셜 기능은 전부 v2 이후.
- **P8. HUFS 스크래핑 ToS** — v1.0 공개 스토어 출시 **전에** 반드시 해결. 단, robots.txt 준수나 저빈도 호출은 **ToS를 대체하지 않음** (robots.txt는 크롤러 규약, ToS는 별도 계약). 한국 법상 fair use 개념도 미국처럼 폭넓지 않음. 따라서 (a)가 주경로, (b)는 리스크 축소책에 불과:
  - **(a) 주경로:** 학사지원처/정보전산처에 학생 사이드프로젝트 명분 서면 문의, 명시적 허가 확보. 거부/무응답 시 B 진행 여부 재검토.
  - **(b) 리스크 축소 (보조):** User-Agent에 프로젝트명/연락처 명시 + 저빈도 호출(하루 1-4회) + 데이터 출처 표기 + 외대 요청 시 즉시 중단 커밋먼트.
  - v0.1 비공개 테스트 단계는 리스크 낮지만, 공개 출시 전 (a) 완료가 원칙.
- **P9. v1 언어 = 한국어 전용** — UI/복사는 한국어만. 코드/주석/커밋/README/기술블로그는 영어 (국제 포트폴리오 어필). i18n 레이어는 v2에서 영어 UI 추가 시 도입.

## Cross-Model Perspective (Claude 서브에이전트 2차 의견)

독립 서브에이전트(프레시 컨텍스트, 이 세션 미열람)의 cold read:

**Steelman:** "사용자는 버스 앱을 만드는 게 아니라 **자신이 AI 에이전트로 출시까지 해낸 과정의 케이스 스터디**를 만드는 것. README, 아키텍처 다이어그램, CI 뱃지, '48시간 동안 혼자 + Claude Code로 30커밋' 같은 스토리가 기능 깊이보다 중요. 한 개의 delightful interaction 있는 MVP가 feature-complete v1보다 강하다."

**핵심 인사이트:** 사용자의 "B" 답변 ("Flutter 배우고 + 포트폴리오 + AI 에이전트팀 배포까지") = 브리프의 전부. 청중은 외대생이 아니라 **리크루터/면접관**. 모든 결정이 이 프레임 하에서 재배치됨.

**도전한 전제:** P1 공개 스토어 출시가 포트폴리오 KPI에 필요한가 — 애플 리뷰 비결정적 대기가 48h 스토리 훼손. 해결: 2단계 출시로 수정 (채택).

**추천 48h 타임박스:**
- 0-8h: CI/CD 완성 (초록 뱃지 커밋)
- 8-16h: Supabase 스크래퍼 Edge Function + 시드 데이터
- 16-32h: "Will I make it?" 단일 화면 (GPS + 확률 링)
- 32-44h: 디자인 시스템 + Rive 히어로 애니메이션
- 44-48h: 30초 녹화 + README + 아키텍처 다이어그램 + LinkedIn 포스트

**추천 OSS/도구:** `very_good_cli`, `very_good_workflows` (GH Actions 템플릿), `supabase_flutter`, `flutter_map` (OSM 타일), `geolocator`, `rive` 또는 `lottie`, `fastlane match`.

## Approaches Considered

### Approach A: Weekend Sprint (MVP 단일 모놀리스)
단일 Flutter 프로젝트, 단일 화면, 최소 의존성. 48h 안에 TestFlight/Internal 링크 공유.
- Effort: S / Risk: Low / Completeness: 5/10
- 포함: "Will I make it?" 한 화면, GPS, 확률 링, 셔틀 스케줄만
- 스킵: 광역버스 API, 클린 아키텍처, 디자인 시스템 풀 세팅

### Approach B: Monorepo Production (풀 아키텍처)
Melos 모노레포, Riverpod/freezed/dio, Supabase + Vercel API 키 프록시, 디자인 시스템, Sentry, 공개 스토어 출시.
- Effort: L (2-4주) / Risk: Medium / Completeness: 9/10
- 포함: "Will I make it?" + 전체 시간표 + 즐겨찾기 + 다크모드 + 광역버스 환승 + 공개 스토어
- 면접 자산 최대화

### Approach C: Flutter + Rust 공유 코어
B와 동일 + 확률 계산 로직을 Rust로 작성 후 `flutter_rust_bridge`로 호출.
- Effort: XL / Risk: High / Completeness: 8/10 (오버엔지니어링)
- 기각 이유: 이 프로젝트 KPI(AI 에이전트 풀스택)에서 벗어남. Rust가 필요할 만큼 복잡한 로직 없음 (~20줄).

## Recommended Approach

**A → B 연속 실행.**

두 스토리 다 확보: "48h one-shot with AI agents" + "프로덕션 레벨 모노레포". A의 모놀리스 코드는 B로 리팩터링되면서 **"어떻게 단일 화면 MVP를 스케일 아웃했는가"**라는 제3의 내러티브도 자연스럽게 발생. 면접에서 가장 풍부한 이야깃거리.

### Milestone A (v0.1, 코딩 48h + 승인 대기 1-7일)

**성공 기준 (시간 축 분리):**
- **Day 0-2 (코딩 48h):** 아래 체크리스트 A 내 "코딩 결과물" 전부 완료
- **Day 2-10 (승인 대기):** Apple Developer 계정 본인인증 + TestFlight External Review 병렬 진행
- **Day 3-10:** 링크 확보 → README/LinkedIn 업데이트

"48시간" = 코딩 세션, "v0.1 완성" = 외부 승인 완료. 혼동 금지.

**리포지토리:** 신규 생성 (`bus-finder-flutter` 또는 `will-i-make-it`).

**디렉토리 구조 (A 단계, 단일 모놀리스):**
```
bus-finder-flutter/
├── .github/workflows/
│   ├── ci.yml              # lint + test
│   └── deploy.yml          # TestFlight + Play Internal
├── ios/ android/           # 기본 생성
├── lib/
│   ├── main.dart
│   ├── will_i_make_it_screen.dart
│   ├── models/             # schedule, stop
│   ├── services/           # supabase client, location
│   └── widgets/            # probability_ring, fallback_card
├── pubspec.yaml
├── fastlane/               # match, deliver, supply
└── README.md               # 아키텍처 다이어그램 포함
```

**Supabase 초기 설정 (A 단계):**
- 테이블 1개: `shuttle_schedules` (id, route, stop_name, departure_times jsonb, day_of_week, valid_from, valid_to)
- Edge Function: `scrape_shuttle_schedule` (pg_cron으로 매일 06:00 KST 실행, HUFS 홈페이지 cheerio로 파싱)
- 시드: 기존 React 앱의 하드코딩된 2026년 1학기 시간표

**의존성 (A):**
- `supabase_flutter`
- `geolocator`
- `rive` 또는 `lottie`
- `intl` (한국어 로케일)

**애플/플레이 세팅 (상세 절차):**

**Apple 쪽 (가장 시간 소요):**
1. Apple Developer Program 가입 — **개인 계정 신원 확인 1-3일 소요** (실패 시 재제출). **Day 0에 먼저 시작, 코드 작업과 병렬**.
2. App Store Connect에서 새 앱 등록 (Bundle ID 예: `com.yourdomain.busfinder`)
3. App Store Connect API Key 생성 (`AuthKey_XXX.p8` 다운로드) → Fastlane에서 사용
4. Fastlane `match` 초기화:
   - 별도 private GitHub 레포(`bus-finder-certs`) 생성
   - `fastlane match init` → git storage 선택 → 레포 URL 입력
   - `fastlane match appstore` → 인증서/프로비저닝 프로파일 자동 생성 & 암호화 커밋
   - `MATCH_PASSWORD` (암호화 비밀번호)를 GH Actions 시크릿에 등록
5. TestFlight External Testing 그룹 생성 → Public Link 활성화 (첫 빌드 업로드 후 External Review 1-3일 소요)

**Google 쪽 (빠름):**
1. Google Play Console 가입 ($25 일회성) — 승인 즉시
2. 새 앱 생성, Internal Testing 트랙 설정 (심사 없음, 업로드 즉시 링크 활성)
3. 서비스 계정 생성 (`google-play-service-account.json`) → GH Actions 시크릿에 base64 등록
4. Fastlane `supply init` → 메타데이터 스켈레톤 생성

**GH Actions 시크릿 (전체):**
```
APPLE_API_KEY_ID       # App Store Connect API Key ID
APPLE_API_ISSUER_ID    # Issuer ID
APPLE_API_KEY_CONTENT  # .p8 파일 내용 (base64)
MATCH_PASSWORD         # match 암호화 비밀번호
MATCH_GIT_TOKEN        # bus-finder-certs 레포 접근 PAT
PLAY_JSON_KEY          # 구글 서비스 계정 JSON (base64)
KEYCHAIN_PASSWORD      # macOS 러너 keychain 임시 비번 (Fastlane match 사용)
```

**첫 시도 시 흔한 함정 (에이전트팀에 미리 알릴 것):**
- macOS 러너에서 `match` 사용 시 keychain 잠금 해제 필요 (`security unlock-keychain` 단계)
- Xcode 자동 서명(automatic signing) 끄기 — Fastlane match는 manual signing 전제
- Provisioning Profile 매칭 실패 → `fastlane match nuke` 후 재생성 (마지막 수단)

### Milestone B (v1.0, 2-4주)

**Melos 모노레포로 리팩터링:**
```
bus-finder-flutter/
├── melos.yaml
├── packages/
│   ├── bus_api/            # 광역버스 공공 API 클라이언트 (Vercel 프록시 통해)
│   ├── shuttle_data/       # Supabase 연동 + 셔틀 모델/리포지토리
│   ├── probability/        # "Will I make it?" 순수 계산 로직 (테스트 100%)
```

## What I noticed about how you think

세션 동안 본인이 "스타트업 모드"를 골랐지만, 압박 질문 2번째 라운드에서 바로 "B (Flutter 배우고 + 포트폴리오)"로 솔직하게 피봇한 게 인상적이었습니다. 많은 사람이 체면 때문에 끝까지 "스타트업 만듭니다"로 밀고 가는데, 당신은 "목적이 달라요"라고 재구성했어요. 이건 현실 파악 능력 + 자기 인지력 신호입니다. 그대로 유지하세요.

두 번째 — 셔틀 데이터 전략 질문에 "현재는 외대 홈페이지 페이지 스크래핑으로 진행했었어. 광역버스 API를 따와서 진행하고 셔틀은 간단하니까 스크래핑해도 괜찮을 것 같아"라고 답한 부분. 이게 **엔지니어의 답변**이에요. 해결된 부분(공공 API)과 미해결 부분(셔틀)을 분리했고, 복잡한 곳(공공 API)엔 제대로 된 도구를, 간단한 곳(셔틀)엔 적절한 수준의 해결책(스크래핑)을 매칭했어요. 이 판단력이 코드를 넘어 제품 전체에 스며들도록 하세요.

세 번째 — Flutter 선택 이유를 고를 때 "B" (솔직)를 고른 뒤 바로 "바이브코딩 에이전트팀으로 앱 배포까지 가고 싶다"고 실행 프레임을 구체화한 것. 많은 러너들이 "배우고 싶다"에서 멈추는데, 당신은 바로 "실행 경로"까지 그렸어요. 그래서 이 설계 문서가 **러닝 플랜**이 아니라 **출시 플랜**이 될 수 있었습니다.

네 번째 — P5 위젯 연기 결정. 제가 미는 "가장 쿨한 버전"을 스코프 기준으로 쳐낸 건 스코프 관리 감각이에요. 포트폴리오 제품의 가장 흔한 실패는 다 끼워넣다가 아무 것도 완성 못 하는 것. 한 번 연기하는 걸 경험했으니, 다음 번에 "이거도 넣자" 유혹이 올 때 같은 감각을 꺼내세요.
